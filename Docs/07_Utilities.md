# 07_Utilities.md

## Overview
This module exposes cryptographic and network exploration tools. It allows the user to prove their identity by cryptographically signing messages with their node's private key and to verify signatures from others. It also includes a search tool to explore the Lightning Network graph, which is enhanced with special logic to support the local test environment.

---

## 1. sign_message

**Description:**  
This API allows the user to create a digital signature using their node's distinct private key. It wraps the Core Lightning `signmessage` RPC command. You provide a text string (e.g., "I verify that I own this node"), and the function returns a cryptographic signature (zbase32 encoded). This is commonly used for authentication—proving ownership of a specific Node ID to a third-party service without revealing the private key itself.

**Parameters:**

- **message (string):** The text content to sign.

**Returns:**

- **signature:** The resulting digital signature.
- **recid:** Recovery ID (technical metadata).
- **zbase32:** The standard encoded signature string.

---

## 2. verify_message

**Description:**  
This function checks the validity of a signed message from another user. It takes a message and a signature, then asks the node (`checkmessage` RPC) to verify if the signature was mathematically generated by the public key associated with the message. A unique feature of this API is that if the signature is valid, it checks the recovered Public Key against the known local nodes (Node 4, 5, or 6). If it matches one of them, it injects a friendly `_alias` into the response (e.g., "Hub" or "Merchant"), making it easier to see who signed the message.

**Parameters:**

- **message (string):** The original text.
- **signature (string):** The signature to verify.

**Returns:**

- **verified (boolean):** `true` if valid.
- **pubkey:** The Node ID of the signer.
- **_alias (optional):** The friendly name of the signer if recognized.

---

## 3. network_node_search

**Description:**  
This API allows the frontend to search the global Lightning Network graph for other nodes. It accepts a search query and filters the `listnodes` database by Alias (fuzzy match) or Node ID (prefix match). Crucially, it contains a **Search Enrichment** block designed for the development environment: since local Regtest nodes often don't advertise public IP addresses, this function manually injects the correct `127.0.0.1` and port information for the known nodes (Node 4, 5, 6) if missing. This ensures that search results in the test environment are actually clickable and connectable.

**Parameters:**

- **query (string):** Text to search for (e.g., "flamingo" or "03a...").
- **limit (number, optional):** Max results to return.

**Returns:**

- **data (array):** List of matching node objects with addresses.
