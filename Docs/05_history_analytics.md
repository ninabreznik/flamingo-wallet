# History & Analytics Module

## Overview

This module provides a comprehensive view of the node's financial history. Because a Lightning node operates across two distinct ledgers, the **Bitcoin Blockchain (On-Chain)** and the **Lightning Network (Off-Chain channels)**, this module aggregates, normalizes, and sorts data from both sources.

It exposes high-level unified feeds for UI consumption as well as low-level raw access to underlying RPC data for detailed inspection and debugging.

---

## 1. `get_transaction_history`

### Description

Primary **Activity Feed** API for the application. It executes a unified aggregation process (`_fetch_unified_history`) that pulls data from three sources:

- Bitcoin Core (`bitcoind`) — on-chain transactions
- Core Lightning `listinvoices` — incoming Lightning payments
- Core Lightning `listpays` — outgoing Lightning payments

The function normalizes all records into a common structure, converts amounts to millisatoshis, and sorts results by timestamp (newest first). This enables the frontend to render a single chronological feed regardless of payment layer.

### Parameters

- `userId` (`string`, optional)
  - Node to query
  - Defaults to `node4`

- `limit` (`number`, optional)
  - Maximum number of records to return
  - Default: `50`

### Returns

- `data` (`array`)
  - Normalized transaction objects containing fields such as:
    - `type`
    - `amount_msat`
    - `status`
    - `timestamp`

## 2. `get_transaction_summary`

### Description

Generates an aggregate financial snapshot across the node’s entire history. The API iterates through the unified transaction log to compute cumulative metrics spanning both Bitcoin and Lightning layers.

Calculated values include:

- Total funds received
- Total funds sent
- Total network fees paid
- Net balance flow

This data is typically used to populate dashboard-level analytics and summary widgets.

### Parameters

- `userId` (`string`, optional)
  - Node to query

### Returns

- `total_received_msat`
  - Total incoming funds

- `total_sent_msat`
  - Total outgoing funds

- `total_fees_paid_msat`
  - Total fees paid across all transactions

- `net_balance_msat`
  - Net flow (received − sent − fees)

- `tx_count`
  - Total number of transactions processed

---

## 3. `get_daily_transactions`

### Description

Filtering utility designed for charts, graphs, and calendar views. It accepts a specific date and filters the unified transaction history to return only activity that occurred on that day.

The function internally converts and matches Unix timestamps against the requested calendar date, allowing precise per-day auditing of spending or income.

### Parameters

- `date` (`string`)
  - Target date in `YYYY-MM-DD` format (e.g., `2025-10-27`)

### Returns

- `data` (`array`)
  - Transactions that occurred on the specified date

---

## 4. `get_transaction_type`

### Description

Lookup utility that determines the nature of a transaction given only its identifier. The function searches the unified history for a matching:

- Bitcoin `tx_id`, or
- Lightning `payment_hash`

If found, it returns the transaction type along with full details. This is primarily used by the frontend to route users to the appropriate detail view (e.g., block explorer vs. Lightning proof screen).

### Parameters

- `tx_id` (`string`)
  - Transaction identifier to search for

### Returns

- `type`
  - `BTC` or `LN`

- `details`
  - Full normalized transaction object

---

## 5. `list_btc_transactions`

### Description

Provides direct access to the **raw Bitcoin transaction history** from the underlying node. This API wraps the `listtransactions` RPC command and returns unmodified blockchain-level data.

Unlike the unified history, this output includes low-level details such as:

- Block indices
- Raw confirmation counts
- Change outputs

Primarily used for debugging or advanced inspection views.

### Parameters

- `count` (`number`)
  - Number of transactions to fetch

- `skip` (`number`)
  - Pagination offset

### Returns

- `data` (`array`)
  - Raw Bitcoin transaction objects

---

## 6. `list_ln_invoices`

### Description

Returns the complete list of Lightning invoices generated by the node directly from Core Lightning’s database.

Unlike the unified history feed, this API includes **all invoices**, including:

- Pending
- Expired
- Unpaid

This is essential for "Payment Requests" or invoice management screens where visibility into unpaid or expired invoices is required.

### Parameters

- `userId` (`string`, optional)
  - Node to query

### Returns

- `data` (`array`)
  - Raw invoice objects (e.g., `bolt11`, `payment_hash`, `status`)

---

## 7. `get_tx_details_from_btcnode`

### Description

Fetches granular technical details for a specific **on-chain** Bitcoin transaction. This API wraps the `gettransaction` RPC call, which exposes information not available in summary lists.

Returned details may include:

- Block hash
- Time received
- Inputs and outputs
- Raw transaction hex

Triggered when a user requests a deep inspection of a Bitcoin transaction.

### Parameters

- `tx_id` (`string`)
  - Bitcoin transaction hash

### Returns

- `data` (`object`)
  - Detailed Bitcoin transaction information

---

## 8. `get_tx_details_from_lnnode`

### Description

Retrieves detailed information about a specific **Lightning payment** by querying the node’s invoice database using the provided payment hash.

This function is commonly used to display proof-of-payment details or to verify exact amounts received versus requested for an invoice.

### Parameters

- `payment_hash` (`string`)
  - Lightning payment identifier

### Returns

- `data` (`object`)
  - Detailed Lightning invoice object

---

## 9. `log_transaction`

### Description

Currently a placeholder endpoint. Intended for future use as a manual or external transaction logging mechanism.

Potential future use cases include:

- Logging cash transactions
- Storing metadata-only records
- Manual adjustments or annotations

At present, it simply confirms that the logging endpoint is reachable.

### Parameters

- None

### Returns

- `message`
  - Confirmation string
