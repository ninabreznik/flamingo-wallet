# 08_Admin_Debug.md

## Overview
This module contains powerful tools designed for Developers and Admins running the Flamingo Wallet in a simulated or Regtest environment. These APIs control the "simulation god mode," allowing you to reset the entire world, force specific network topologies (routing paths), and simulate traffic to test the UI. **Warning:** These functions are destructive and typically meant for a controlled testnet environment.

---

## 1. admin_reset_world

**Description:**  
Known as the "Nuclear Option," this API performs a complete cleanup of the ecosystem. It iterates through every channel on both the User Node (node4) and the Hub (node5). If a channel is active, it issues a close command; if a channel is dead/on-chain, it disconnects the peer. Specifically for the Regtest environment, it effectively "buries" these dead channels by automatically mining 10 Bitcoin blocks immediately after closing them, forcing the transactions to confirm and clearing the state. It returns a log of all actions taken.

**Parameters:**  
None.

**Returns:**

- **message:** Confirmation of cleanup.  
- **logs:** Array of specific actions (closed channels, mined blocks).

---

## 2. admin_setup_hub

**Description:**  
This function automates the creation of the backend routing topology required for the Flamingo demo. It ensures that the Hub (Node 5) has an active, funded channel open to the Merchant (Node 6). It checks the existing connection state; if no channel exists, it opens one with 5,000,000 sats. If the channel exists but is not yet active (e.g., `CHANNELD_AWAITING_LOCKIN`), it reports the status and instructs the admin to mine blocks. This setup is required to test multi-hop routing (Node 4 -> Node 5 -> Node 6).

**Parameters:**  
None.

**Returns:**

- **message:** Status of the route.  
- **txid:** Transaction ID of the channel funding (if created).

---

## 3. verify_revenue

**Description:**  
This is a full system integration test designed to verify that the Hub (Node 5) is earning routing fees. It executes a complex sequence:

- Node 6 (Merchant) generates an invoice.  
- Node 4 (User) pays that invoice (forcing the payment to route through Node 5).  
- The system waits and then queries Node 5's forwarding logs.  
- It verifies that the payment succeeded and explicitly checks that Node 5 collected a non-zero fee (`fee_msat > 0`).  

This confirms the business model logic is working correctly.

**Parameters:**

- **amount_msat (number):** Amount to test with (default 10,000).

**Returns:**

- **verification:** `'PASSED'` or `'FAILED'`.  
- **fee_earned_msat:** The exact fee collected by the Hub.

---

## 4. simulate_incoming_payment

**Description:**  
This tool is used to test the "Receive" UI without needing a real second wallet. It commands the Hub (node5) to pay an invoice generated by the User (node4). A key feature of this API is its **Auto-Healing Logic**: if the payment fails (e.g., because there is no route or insufficient liquidity), the code catches the error, automatically opens a new channel from Node 5 to Node 4, mines the required blocks to confirm it, and then retries the payment. This ensures the test almost always succeeds, even if the network was previously disconnected.

**Parameters:**

- **bolt11 (string):** The invoice string to pay.

**Returns:**

- **message:** Confirmation of payment.  
- **result:** The payment details or error logs.
